## Six-Lens Plan

### 1. Company Genome Lens
**Future state**
- Workspace, company, project, decision, capability, department, studio, artifact, experiment, and run objects all have canonical ownership and lifecycle rules.

**12-month build**
- Add a department registry on top of current company and capability records.
- Tie each studio to one department and one execution track.
- Require one scorecard and one operating cadence per department.

**Metric**
- `% of active work mapped to a canonical company/project/decision/department path`

### 2. Evidence Graph Lens
**Future state**
- Duplicate information is resolved through canonical nodes, evidence tiers, and clear object references rather than multiple overlapping docs.

**12-month build**
- Build duplicate-surface inventory.
- Define canonical source rules for strategy, research, templates, and studio memory.
- Promote best artifacts into reusable resource libraries by department.

**Metric**
- `% of core capabilities with a canonical evidence path and no unresolved duplicate owner`

### 3. Cognitive Studios Lens
**Future state**
- Every department studio runs the eight-layer cognitive loop: perception, diagnosis, value model, concept, challenge, experiment, production, memory.

**12-month build**
- Standardize department packages.
- Finish Consumer User Studio as the reference operating template.
- Expand Marketing, Product, Engineering, and Job Studio into full department-grade studios.

**Metric**
- `% of department studios with one callable entrypoint, process pack, and memory loop`

### 4. Trust and Governance Lens
**Future state**
- New departments and products inherit accessibility, privacy, security, policy, and claims gates by default.

**12-month build**
- Define release gates for high-trust work.
- Put Trust & Governance Studio into every build sequence.
- Attach approval and audit traces to decisions and publishes.

**Metric**
- `% of launches with trust, accessibility, and security checks attached`

### 5. Operator Console Lens
**Future state**
- The dashboard is a state-aware neural operator layer that senses events, routes departments, proposes next actions, and records execution.

**12-month build**
- Move from widget dashboard to object graph.
- Add event triggers:
  - stale evidence
  - failing tests
  - blocked stage gate
  - overdue decision
  - drifting capability score
- Attach next-best-action routing to each event.

**Metric**
- `median time from signal to routed next action`

### 6. Memory and Evaluation Lens
**Future state**
- The OS compounds. Every studio run, decision, experiment, and launch improves the next one through writeback and evaluation.

**12-month build**
- Add writeback standards for all department studios.
- Add evaluation hooks for diagnosis quality, implementation realism, and outcome quality.
- Tie experiments and postmortems back to capability maturity.

**Metric**
- `% of major runs that generate memory, eval, and artifact writeback`
