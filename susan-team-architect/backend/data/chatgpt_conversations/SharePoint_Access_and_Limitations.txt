What exists today
Things you can often do with only site/list access
If you have edit rights or site-owner-level rights on a SharePoint site, the main “build without admin” options are usually:


Power Apps custom forms for SharePoint lists


You can customize a SharePoint/Microsoft Lists form in Power Apps when you have write access to the list and permission in Power Platform to create/update the form. This is one of the cleanest ways to add logic, validation, conditional UI, and lightweight workflow behavior without full tenant admin rights. Microsoft Learn+1




List/view/column JSON formatting


This is not full code, but it does let you create custom UI behavior, conditional rendering, inline editing scenarios, buttons, gallery-like layouts, and richer list experiences. It is much safer and more likely to be allowed than custom script. Microsoft Learn+1




Power Automate flows tied to SharePoint


In many tenants, users with the right licenses can create flows on top of SharePoint lists/libraries they can access. This is good for notifications, approvals, item updates, file routing, and document workflows. Whether it works in your tenant depends on Power Platform governance, DLP rules, connector policy, and environment permissions. Microsoft’s Power Platform docs make clear that app/flow creation is permissioned separately from raw SharePoint access. Microsoft Learn+1




External code that runs as you, against content you can already access


You can absolutely write code locally in VS Code, Cursor, Windsurf, Visual Studio, PowerShell, Python, or Node and call SharePoint/Graph APIs as your own identity, but that only works within the permissions you already have. Delegated permissions act on behalf of the signed-in user and do not let the app reach beyond what that user can access. Microsoft Learn




Things that usually hit an admin wall
These are the common blockers:


SharePoint Framework (SPFx) / packaged custom apps


Deploying SPFx solutions requires an app catalog, and Microsoft documents that app catalogs are managed through the SharePoint admin center by org admins. Tenant-scoped deployment is explicitly an app-catalog deployment path. Microsoft Learn+2Microsoft Learn+2




Custom script / classic script injection


Whether custom script is allowed is controlled by the SharePoint admin center. If it’s blocked, you do not get to turn it on yourself from normal site permissions. Microsoft Learn+2Microsoft Learn+2




App registrations / broader Graph or SharePoint app permissions


Admin consent is required for many app permissions. Microsoft’s consent docs and Graph permission model make that explicit. Selected scopes and app-only patterns also require an admin-mediated setup path. Microsoft Learn+3Microsoft Learn+3Microsoft Learn+3




Site collection app catalog enablement


This is a useful compromise because it narrows deployment to one site collection, but Microsoft still says tenant admins must enable the site collection app catalog first. After that, site collection admins can deploy solutions within that scope. Microsoft Learn+1




Direct answer to your two questions
1) Are there programs that will allow me to write code with only SharePoint edit/site access?
Yes, but with limits.
Conservative


Use Microsoft Lists + JSON formatting + Power Apps custom forms + Power Automate.


Best for: forms, workflow, approvals, light logic, UI improvements.


Pros: lowest friction, lowest security concern, most likely to be allowed.


Cons: not “real arbitrary code”; can get clunky at scale.


Confidence: high. Microsoft Learn+2Microsoft Learn+2


Base


Use local coding tools like VS Code/Cursor/PowerShell/Python/Node and authenticate as yourself to SharePoint/Graph for the sites and lists you already have access to.


Best for: migration utilities, reporting, controlled CRUD on lists/files, automation run from your machine.


Pros: real code, faster development, no need to deploy into SharePoint UI.


Cons: limited to your existing rights; app registration or consent may still become a blocker depending on auth method.


Confidence: medium-high. Delegated access can work, but exact auth patterns depend on tenant policy. Microsoft Learn+1


Aggressive but still legitimate


Ask for a site collection app catalog to be enabled for one site collection, or for an admin-approved app registration with narrow scopes like selected site/list access.


Best for: real custom components without going tenant-wide.


Pros: biggest jump in capability with a smaller blast radius than full tenant enablement.


Cons: still requires admin involvement; not something you can self-grant.


Confidence: high that this is the right “middle path,” low that you can do it alone. Microsoft Learn+2Microsoft Learn+2


2) How can I bypass admin rights if necessary?
You can’t do that legitimately, and I’m not going to help with bypassing or evading access controls.
What you can do instead is reduce the ask so an admin is more likely to approve it:


Request one-site-only enablement, not tenant-wide access.


Ask for a site collection app catalog, not a tenant app catalog deployment.


Ask for selected scopes or resource-specific consent, not broad Sites.ReadWrite.All.


Ask for a sandbox/dev site where custom script is allowed, instead of production-wide changes.


Ask for a service account or approved app registration with narrow delegated/app-only scope to one site/list/library. Microsoft’s docs support selected-scope and site-targeted permission patterns, but they still require proper consent and assignment. Microsoft Learn+3Microsoft Learn+3Microsoft Learn+3


Recommendation
Go with the base path:


Start with Power Apps + Power Automate + JSON formatting for anything that lives inside the SharePoint experience.


In parallel, use local code against SharePoint/Graph as your own identity for reporting, transformation, and controlled automation on content you already can access.


If you hit UI/component limits, ask IT for a site collection app catalog on one site or a narrow app registration rather than asking for broad admin rights.


Why this is the best route:


It gets you moving immediately.


It avoids a dead-end “can I secretly get around admin?” path.


It gives IT a smaller, defensible security request if you need more later. Microsoft Learn+3Microsoft Learn+3Microsoft Learn+3
