## Executive summary

Fitness app UX since 2018 has been converging on three mutually reinforcing directions: conversational coaching (chatbots/virtual coaches), “just-in-time” coaching timed to user context, and richer real-time telemetry (heart rate, reps, form, weights) displayed in a way that stays glanceable under exertion. Evidence syntheses show conversational agents can improve physical activity outcomes, but effects vary and depend heavily on usability, personalization, and when/why messages are delivered. citeturn0search0turn0search1turn6search1

Your target concept—**a motion-narrative, text-message workout dialogue where a coach “is with you” while a user can always see weights and reps**—is best served by a **mantle-and-sidecar layout**:

- **Mantle (always visible):** a pinned “Lift Card” (or “Set HUD”) that never yields its space and always shows *current exercise, set number, target reps, logged reps, load, rest timer, and next action*.  
- **Sidecar (contextual):** a coach message thread that is **collapsed during the set** (to minimize reading burden) and **expands during micro-pauses** (setup, rest, transitions), with motion used sparingly to reinforce state changes rather than decorate. This aligns with platform motion guidance to avoid adding motion to frequent interactions. citeturn0search3turn2search3

To make “cognitive architecture coaching AI agents” concrete in UX, treat the coach as an **agent with an explicit perception→plan→act loop** (telemetry + context as “perception”; set-level decisions as “planning”; short prompts as “actions”). Recent AI-agent architectures formalize modular memory + decision loops (useful for coaching continuity), and “self-aware / metacognitive” layers add monitoring and failure prediction (useful for safety, uncertainty, and not interrupting at the wrong moment). citeturn5search1turn6search0turn6search8turn6search1

Just-in-time adaptive interventions (JITAIs) formalize delivering prompts when a user is most receptive and when a prompt can actually change behavior. Recent protocols and studies continue to expand JITAI implementations using wearables and context (e.g., tailoring prompts based on real-time and historical data), reinforcing that *timing/decision rules are core product IP*, not copywriting. citeturn3search13turn3search25

### Cognitive architecture coaching agents and “self-aware” layers

Recent “cognitive architecture for language agents” work proposes modular memory + structured action spaces + decision loops, mapping cleanly to a workout coach that must (1) remember user context, (2) choose among actions like “prompt”, “suggest weight”, “stay silent”, “ask to confirm”, and (3) react to new telemetry. citeturn5search1turn5search24

## UI patterns and interaction flows for a motion-narrative coach thread with always-visible weights and reps

### Core layout pattern: mantle-and-sidecar

**Goal:** preserve *continuous visibility* of weights/reps while enabling a coach “presence” that feels live.

**Recommended layout primitives**
- **Pinned Lift Card (mantle):** occupies ~30–40% of vertical space (phone portrait) or a fixed top/side region (tablet/landscape). Never scrolls away.  
- **Coach Thread (sidecar):** a message list that can be (a) collapsed to a single “coach chip” during sets, (b) expanded during rest, and (c) fully opened when the user explicitly wants narrative detail.  
- **Action Rail:** one-thumb reach controls: **Log set**, **+/- reps**, **+/- weight**, **Start rest**, **Pause**, **Skip**, **Edit exercise**.

This pattern is intentionally biased toward glanceability because motion + reading while moving can degrade performance and comprehension; research on mobile reading while walking shows measurable drops in reading performance, and mitigation approaches such as automatic font adjustments help reduce that degradation. citeturn2search27turn1search38

### Interaction flow: when the coach speaks vs. stays silent

A workout is naturally segmented into **setup → set → immediate post-set → rest → transition**. Your UI should treat these as different “attention budgets.”

- **Setup (high receptivity, low exertion):** coach offers one short cue (goal, form reminder, or target), then defers.  
- **Set (lowest receptivity for reading):** coach *defaults to silent text*; optional audio/haptic cues only for critical events (time cap, stop cue, safety).  
- **Immediate post-set (“log moment”):** quick confirm: “Logged 8 reps @ 135?” with tappable corrections.  
- **Rest:** narrative expands: encouragement, next-set suggestion, technique hint, or micro-lesson; keep it skimmable.  
- **Transition:** coach preview + “next up” card; minimal motion.

This mirrors how hands-free and hybrid coaching concepts separate real-time support from heavier explanation, and aligns with platform guidance to keep frequent interactions subtle. citeturn10search28turn0search3

### Concrete UI patterns

#### Pattern: Pinned Lift Card with “log loop” microinteractions

**Pinned Lift Card contents (always visible)**
- Exercise name + variant
- Set x of y
- Target reps (or time)
- **Weight** (large) + quick adjust
- **Reps** (large) + quick adjust
- Rest countdown (when active)
- Next exercise preview (collapsed)

Fitbod’s Apple Watch and rest-timer docs support the expectation of set logging plus sound/haptic rest cues, illustrating a proven workflow for “log quickly, stay focused.” citeturn8search16turn8search0

**Microinteraction recommendation (organic, but restrained):**
- When user taps **Log set**, apply a *brief* confirmation animation (e.g., card “settles” into place) rather than a large directional slide, aligning with guidance to avoid heavy motion for frequent interactions. citeturn0search3turn1search10

#### Pattern: Coach Thread as “stateful narration,” not chatty banter

Treat each coach message as one of four **message types** (visualized as subtle status chips):
1. **Directive** (do now): “Start set—8 reps. Control the eccentric.”
2. **Confirm** (log/check): “I saw 8 reps @ 60 lb. Correct?”
3. **Reflect** (after set): “Tempo looked steady. RPE?”
4. **Plan** (next): “Next set: +2.5 lb, aim 6–8.”

This maps to cognitive agent loops: perceive → decide → act → reflect. citeturn5search1turn5search2

#### Pattern: Telemetry “confidence” and metacognitive transparency

If you auto-detect reps/weights (camera/BLE), introduce a **confidence affordance**:
- “8 reps (high confidence)” vs “~8 reps (check)”  
- A single-tap correction UI

This is where “self-aware architecture” matters: metacognitive layers explicitly monitor uncertainty/failure risk and can switch the UX into “ask/confirm” mode. citeturn6search0turn6search8turn6search7

Tempo’s positioning around real-time feedback and weight recognition shows consumers will accept automated sensing—*but your UX must handle uncertainty gracefully*. citeturn8search33turn3search24

#### Pattern: Pause/skip/adjust that never hides the numbers

A recurring failure in narrative UIs is that overlays cover the critical metric. Use:
- **Persistent metric region** (numbers never occluded)
- **Bottom-sheet coach expansion** that stops above the Lift Card
- **Voice + haptic confirmations** for “pause/skip” where available (reduces screen interaction)

Apple’s workout-app guidance highlights designing around always-on updates and workout metrics surfaces. citeturn7search28turn2search36

## Accessibility, safety, and exertion UX constraints

### Cognitive load and “readability under motion”

Workout logging happens under elevated arousal and sometimes physical motion (standing, pacing, repositioning). Empirical work on mobile reading while walking finds walking degrades performance, and adaptive presentation (e.g., automatic font size adjustment) can mitigate some degradation. citeturn2search27turn1search38

Wearable visualization research similarly emphasizes that relative motion (user moving + display moving) increases workload and reduces reading task performance—supporting a “glanceable first, narrative second” hierarchy. citeturn1search15

**Design implications**
- During sets: do not require reading paragraphs; prefer *single-line cues* or iconography.
- Between sets: allow narrative, but keep it scannable and chunked (1–2 bubbles).
- Offer an accessibility option to increase the Lift Card type scale aggressively; Apple explicitly encourages supporting large text scaling (commonly at least 200% of default). citeturn9search4turn9search0

### Motion sensitivity and reduced-motion compliance

Motion-narrative UI can conflict with vestibular sensitivities if animations are continuous or involve large multi-axis movement. WCAG guidance for “Animation from Interactions” emphasizes allowing motion triggered by interaction to be disabled unless essential, and OS-level “prefers-reduced-motion” patterns operationalize this on modern platforms. citeturn7search3turn1search2turn1search10

Apple’s reduced-motion evaluation criteria call out disabling or replacing potentially problematic motion (parallax, spinning, ongoing motion) when the user setting indicates reduced motion. citeturn0search7

**Practical rule:** keep narrative “alive” via *opacity, scale, and color emphasis* rather than large translations; provide a reduced-motion variant that uses cross-fades and static state changes.

### Audio/haptics as “eyes-free” narrative carriers

Fitbod’s rest timer explicitly uses tone and/or vibration to signal transitions between sets. citeturn8search0  
Apple’s audio-haptics guidance and Core Haptics tooling indicate a platform-supported path for synchronized cues. citeturn0search36turn0search38

**Pattern:** shift the “coach presence” from reading to *ambient cues*:
- haptic “ready” cue at rest end
- short audio “start” cue (optional)
- voice coaching opt-in (especially for users training hands-free), consistent with how Workout Buddy delivers encouragement. citeturn5search0

### Safety framing and exercise guidelines context

If your app is positioning itself as a coach, it will be evaluated (by users and regulators) against mainstream exercise guidance. Global recommendations emphasize adults accumulating weekly aerobic activity and doing muscle-strengthening activities at least 2 days/week; your content library and coaching language should not contradict that baseline. citeturn1search4turn1search20turn1search24

## Technical considerations for real-time set telemetry and agentic coaching

### Data sources and interoperability

You have three practical telemetry tiers:

**Manual-first (fastest to ship):** user logs reps/weight; watch/phone provides timers and haptics (Fitbod-style). citeturn8search16turn8search0

**BLE sensors and machines (interoperable telemetry):**
- Bluetooth SIG’s **Fitness Machine Service (FTMS)** is designed to expose training-related data from fitness machines to clients. citeturn0search2turn0search6  
- Bluetooth SIG’s **Heart Rate Service** is a standard GATT service for heart-rate sensors used in fitness. citeturn7search2turn7search5  

**Camera / CV inference (rep/form):**
- Commercial examples: Tempo describes camera-based joint tracking + form feedback and weight recognition. citeturn8search2turn3search24turn8search33  
- Research examples: real-time smartphone rep counting can run on-device, supporting offline scenarios. citeturn8academia39

### Platform health data rails, permissions, and syncing

On Apple platforms:
- HealthKit requires user permission for reading/writing health data; Apple Platform Security describes permission requirements and protection class for stored Health data. citeturn9search2turn9search13turn9search23  
- Live workout building APIs support constructing workouts incrementally from a workout session (relevant if you mirror/record sessions in real time). citeturn7search0turn7search16  

On Android platforms:
- Health Connect is positioned as a common on-device API for storing/sharing health and fitness data across apps, with explicit guidance to request only needed permissions. citeturn7search1turn9search7turn7search17  
- Google Play restricts access to sensitive health permissions to approved use cases and expects clear user benefit and accurate declarations. citeturn9search3turn7search29

**Implication for your coaching agent:** architect so the agent can function with “local first” workout state even if HealthKit/Health Connect permissions are denied (otherwise your narrative coach risks failing at onboarding).

### Latency, offline mode, and synchronization strategy

For your UI concept, **latency tolerance is asymmetric**:
- The **Lift Card** must update essentially immediately after user actions (log/adjust), or users will mistrust the system.
- The **coach thread** can tolerate slightly higher latency if it never blocks the set flow and if it communicates when it is “catching up.”

If you support wearable always-on experiences, Apple’s workout app guidance explicitly calls out updating workout metrics using timelines in always-on scenarios—useful for designing “persistent numbers” that don’t stutter. citeturn7search28turn2search36

Offline path options:
- Manual logging is inherently offline.
- On-device CV rep counting is feasible per published smartphone systems. citeturn8academia39  
- BLE is local but requires robust reconnect/error recovery (your UX must anticipate partial data).

### Privacy and trust posture for an “always-with-you” coach

A coach thread is intimate: it implies ongoing tracking and memory. Apple states users control what data is stored in Health and what is shared with third-party apps, and HealthKit interactions require permission. citeturn9search16turn9search2  
Google’s ecosystem similarly treats Health Connect data as sensitive and subject to additional requirements. citeturn9search14turn9search7

**Self-aware architecture as privacy UX:** a metacognitive layer can enforce rules like:
- do not store raw sensor streams unless needed
- store derived summaries by default
- explicitly mark what the coach “remembers” and allow deletion  
This aligns with the broader “agentic metacognition” framing of monitoring and failure prediction—here, “failure” includes privacy overreach. citeturn6search0turn6search8

## Recommended interaction patterns, component specs, microcopy, and prototype roadmap

### Component specs for an organic, motion-narrative workout dialogue

**Component: Lift Card (pinned)**
- **Typography:** use platform typography guidance and support dynamic type/large text. Apple emphasizes typographic hierarchy and legibility, and its app accessibility criteria recommend supporting substantial text enlargement. citeturn9search0turn9search4  
- **Hierarchy:** “Weight” and “Reps” should be the largest elements; everything else is secondary.
- **Touch targets:** 1-tap controls adjacent to numbers; avoid hidden gestures during exertion.

**Component: Coach Thread (expandable)**
- Default collapsed during sets; expands during rest.
- Message bubbles capped at ~120 characters by default; use structured chips for numeric feedback (RPE, tempo, confidence).
- “Coach is typing” indicator should not be a constant animation; use minimal pulses and honor reduced motion.

**Component: State Chips**
- Examples: SET LIVE, REST 01:30, CHECK REPS, SENSOR LOST, LOW CONFIDENCE.
- Chips map cleanly to a cognitive agent action space (observe → choose action → present chip). citeturn5search1turn5search2

**Motion system (tokens, not one-offs)**
- Use a small set of duration/easing tokens (fast/standard/slow) and apply consistently; Material 3 publishes easing/duration guidance and token specs that can serve as a starting point. citeturn2search0turn10search0  
- Provide a reduced-motion variant per OS preference; align with Apple reduced-motion evaluation guidance and WCAG “Animation from Interactions.” citeturn0search7turn7search3

**Implementation options for organic motion assets**
- Lottie for lightweight, cross-platform micro-animations authored in After Effects and rendered natively. citeturn10search1  
- Rive for interactive, state-machine-driven animations that respond to real-time states (e.g., “coach presence” avatar subtly responds to set/rest). citeturn10search2turn10search31

### Sample message-thread microcopy

**Setup**
- Coach: “We’re on Bench Press. Target 6–8. Keep shoulders packed.”

**Set start**
- Coach (chip only): “Brace + drive.”

**Immediate post-set confirm (auto-filled)**
- Coach: “Logged: 8 reps @ 135. Correct?”
- Buttons: YES EDIT REPS EDIT WEIGHT

**Rest narrative (2 bubbles max)**
- Coach: “Nice. Bar speed slowed on reps 7–8.”
- Coach: “Next set: keep 135 if RPE ≥8. Otherwise try 140.”

**Adjustment**
- You tap TRY 140
- Coach: “Locked. Same target (6–8).”

**Uncertainty / self-aware mode**
- Coach: “I’m not sure I caught the last rep. Want to confirm reps before we continue?”
- Buttons: CONFIRM SKIP

This “ask-to-confirm when uncertain” behavior is the UX translation of metacognitive monitoring and reduces the risk of false certainty. citeturn6search7turn6search0

3) How to mix motivational interviewing with loss aversion without making the coach obnoxious
My blunt view: MI should own the tone; loss aversion should own the commitment mechanic.
That split matters. MI works because it is collaborative and autonomy-supportive. Loss aversion works when users feel they are protecting something they earned. Evidence on physical-activity incentives and later analyses suggests loss framing can help, and a later gamification analysis found loss aversion was stronger when the threatened loss was tied to something earned rather than simply endowed by the system. That is compatible with MI only if the user opted into the commitment. BMJ+4PMC+4PubMed+4
So the design rule is:


Use MI in language: ask permission, reflect ambivalence, offer choices, reinforce self-efficacy.


Use loss aversion in state: streaks, plan integrity, “protect your gains,” “save the week,” “keep tomorrow viable.”


Do not use loss aversion in tone: no scolding, no guilt, no body-shame, no “you’re falling behind” during or right after exertion.
