# Founder Company Genome Spec

## Purpose

The company genome is the canonical state model Susan uses to reason about a company before she routes work, opens a studio, or creates a new domain pack.

## Required entities

- company profile
- product lines
- user segments
- jobs to be done
- workflows
- capabilities
- execution tracks
- councils
- datasets
- trust requirements
- KPIs
- experiments
- decisions

## Required company fields

- company id
- name
- domain
- stage
- target market
- product description
- current challenges
- primary jobs to be done
- budget per month
- decision owner
- operating cadence
- primary risks
- current active companies and workstreams

## Genome rules

1. Susan should not treat company identity as only name and category.
2. Every company should have explicit user, value, workflow, and trust definitions.
3. Every company should have execution tracks with owners, outputs, and required evidence.
4. Every company should expose current stage and next stage gate.
5. Every company should have a writeback path into memory, experiments, and decisions.

## Minimum viable company genome

- clear problem statement
- primary user and buyer
- value thesis
- current product surface
- trust constraints
- execution tracks
- operating rhythm
- KPIs
- launch blocker list

## Output contract

When Susan renders a company genome she should return:

- company summary
- stage and current gate
- execution tracks and coverage
- councils and sources
- active risks
- active experiments
- operator metrics
- next actions
