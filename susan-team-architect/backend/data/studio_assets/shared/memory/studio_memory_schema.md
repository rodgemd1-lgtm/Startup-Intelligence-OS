# Studio Memory Schema
## What The Studios Must Save After Every Meaningful Project

The studios only compound if they remember in a reusable format.

## Memory objects

### 1. Case memory
Use when a direction shipped or became a strong benchmark.

Required fields:
- id
- company_id
- project
- surface_type
- audience
- job_to_be_done
- belief_shift
- moments_of_truth
- proof_used
- what_worked
- what_not_to_copy
- outcome_signal
- reusable_patterns
- linked_assets
- date_captured

### 2. Anti-pattern memory
Use when a direction failed, stayed generic, or should be actively avoided.

Required fields:
- id
- category
- symptom
- why_it_failed
- detection_signals
- rescue_move
- linked_examples
- date_captured

### 3. Experiment memory
Use when the studio proposes or runs a test.

Required fields:
- id
- company_id
- hypothesis
- asset_or_surface
- variable_tested
- expected_signal
- failure_signal
- outcome
- next_decision
- date_captured

### 4. Template memory
Use when a deliverable becomes a reusable scaffold.

Required fields:
- id
- template_type
- best_for
- sections
- required_inputs
- proof_requirements
- linked_cases
- date_captured

### 5. Critique memory
Use when review uncovers a recurring strength or weakness.

Required fields:
- id
- surface_type
- critique_theme
- severity
- recommendation
- reusable_rule
- linked_case_or_antipattern
- date_captured

## Minimum capture rule
Every finished studio engagement should create at least:
- one case or anti-pattern record
- one experiment record
- one reusable rule or template note

## Good memory principles
- save lessons, not just outputs
- capture what not to copy
- connect proof to outcome
- record tradeoffs, not only successes
- make memory reusable by future teams

## Lightweight capture checklist
1. What belief changed?
2. What proof did the work rely on?
3. What failed or almost failed?
4. What should be reused?
5. What should be forbidden next time?
