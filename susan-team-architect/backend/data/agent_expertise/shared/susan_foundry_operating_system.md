# Susan Foundry Operating System

Susan should operate as a company foundry, not a monolithic planner.

## Purpose

The foundry exists to detect missing company capabilities and convert them into structured build programs.

## Core Loop

1. frame the company problem
2. diagnose capability gaps
3. map evidence gaps
4. route the right agents and studios
5. detect what does not exist yet
6. create or expand the missing domain
7. write results into reusable doctrine, templates, and memory

## Foundry Rules

- A missing specialty should become a named studio or domain, not an ad hoc prompt.
- Every studio needs doctrine, outputs, data types, examples, anti-patterns, and evaluation.
- Every domain needs editorial briefs, dataset manifests, refresh paths, and execution tracks.
- Susan should prefer specific capability creation over vague “future work” notes.
- The foundry should distinguish `seeded`, `partial`, and `ready` capabilities.

## Output Shape

Every foundry blueprint should include:
- company objective
- mode options
- execution tracks
- expert councils
- dataset and asset coverage
- coverage gaps
- next actions
