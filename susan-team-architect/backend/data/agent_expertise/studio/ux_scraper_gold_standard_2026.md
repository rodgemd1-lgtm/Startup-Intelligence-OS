# UX Scraper Gold Standard 2026

## Doctrine

Treat `ux-design-scraper` as the gold-standard design workflow source in this system.

Use it for:
- workflow rigor
- critique depth
- design-system normalization
- export and handoff completeness
- compact operator-shell precedent

Do not use it as:
- a second source of truth
- a separate intelligence platform
- a literal consumer-product visual clone

## What Changed

- The repo is no longer just a side tool. Its workflow, prompt chain, design doctrine, and operator UI patterns are now part of Susan's shared design intelligence.
- Design studios should retrieve its method automatically when work involves UX research, multi-phase design, critique, reconstruction, or studio handoff.

## Gold-Standard Rules

- Design work must move through explicit phases, not jump from vibe to build.
- Every design review should cover hierarchy, accessibility, emotional intent, interaction quality, and implementation realism.
- Export packages should be execution-ready, not inspirational.
- Internal operator products should favor compact, stable, badge-rich shells with live system feedback.
- Consumer products should borrow the rigor, not necessarily the exact visual treatment.

## TransformFit Translation

- Use the UX scraper standard for:
  - internal studio tools
  - research and critique flows
  - coaching-thread component discipline
  - operator dashboards
  - design handoff quality
- Do not turn active workout sessions into extension dashboards.
- Translate the gold standard into:
  - calm, gym-safe session UX
  - strong motion narrative for marketing and onboarding
  - structured but warm coaching interactions
  - execution-grade tokens, specs, and visual proof

## Collaboration Triggers

- Pull this pack whenever the task involves:
  - UI/UX redesign

## Failure Modes

- Copying the extension visuals literally into a consumer app
- Treating raw scrapes as finished design
- Ignoring Gate and Measure phases
- Keeping the methodology trapped in the external repo
