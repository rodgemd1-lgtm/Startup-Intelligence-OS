## 1. Product Management Frameworks

### Prioritization Frameworks

| Framework | Description | Best For |
|-----------|-------------|----------|
| **RICE** | Reach × Impact × Confidence ÷ Effort | Data-driven prioritization |
| **ICE** | Impact × Confidence × Ease | Quick scoring when RICE is overkill |
| **Kano Model** | Classify features as Must-Have, Performance, or Delight | Understanding customer satisfaction drivers |
| **MoSCoW** | Must have, Should have, Could have, Won't have | Sprint planning, scope negotiation |
| **Weighted Scoring** | Score features against multiple criteria with weights | Complex multi-stakeholder decisions |
| **Opportunity Scoring** | Importance vs Satisfaction gap | Finding underserved needs |
| **Cost of Delay** | Quantify the cost of not building a feature | Time-sensitive decisions |

### Product Discovery Frameworks

| Framework | Description | Resource |
|-----------|-------------|----------|
| **Jobs-to-be-Done (JTBD)** | Understand the "job" customers hire your product to do | Book: *Competing Against Luck* — Clayton Christensen |
| **Design Sprint** | 5-day process: Map → Sketch → Decide → Prototype → Test | Book: *Sprint* — Jake Knapp (Google Ventures) |
| **Shape Up** | 6-week cycles: Shape → Bet → Build. Basecamp's method. | Free book: [basecamp.com/shapeup](https://basecamp.com/shapeup) |
| **Continuous Discovery** | Weekly customer interviews + opportunity solution trees | Book: *Continuous Discovery Habits* — Teresa Torres |
| **Lean Startup** | Build → Measure → Learn cycles. MVP-first approach. | Book: *The Lean Startup* — Eric Ries |
| **Double Diamond** | Diverge → Converge → Diverge → Converge | UK Design Council |
| **Outcome-Driven Innovation** | Focus on measurable customer outcomes, not features | Strategyn methodology |

### PLG Framework

```
ACQUISITION (How users find you)
├── Organic: SEO, content, word-of-mouth, community
├── Viral: Invite loops, collaboration features, shared outputs
└── Product: Free tier, open-source core, freemium

ACTIVATION (First "aha" moment)
├── Time-to-value: How fast do users get value?
├── Onboarding: Guided tours, templates, sample data
└── Milestone: What action correlates with retention?

RETENTION (Keep users coming back)
├── Habit loops: Daily/weekly usage triggers
├── Switching costs: Integrations, data lock-in, workflows
└── Expanding value: Features that compound over time

MONETIZATION (Convert to paid)
├── Usage-based: Pay for what you use (tokens, API calls)
├── Seat-based: Per-user pricing
├── Feature-gated: Free tier → Pro features
└── Hybrid: Combination approaches

EXPANSION (Grow within accounts)
├── Seat expansion: More team members
├── Usage growth: Higher tiers
└── Platform: Add-ons, marketplace
```

### PLG Metrics That Matter

| Metric | What It Measures | Target |
|--------|-----------------|--------|
| **Time to Value (TTV)** | How fast users reach "aha moment" | < 5 minutes ideal |
| **Activation Rate** | % of signups who reach key milestone | > 40% |
| **Natural Rate of Growth (NRG)** | % of revenue from organic/product channels | > 80% for PLG |
| **PQL (Product Qualified Lead)** | Users who hit usage thresholds indicating buying intent | Track conversion rate |
| **Net Revenue Retention (NRR)** | Revenue retention including expansion | > 120% |
| **Viral Coefficient** | Average invites per user × conversion rate | > 1 for virality |

## 5. Technical Architecture for AI Startups

### Stack Recommendations

#### Full-Stack AI SaaS (Recommended Starting Stack)

```
FRONTEND
├── Framework: Next.js 15 (App Router)
├── Language: TypeScript
├── Styling: Tailwind CSS + shadcn/ui
├── State: Zustand or Jotai
└── Hosting: Vercel

BACKEND
├── API: FastAPI (Python) or tRPC (TypeScript)
├── Database: PostgreSQL (Supabase or Neon)
├── Cache: Redis (Upstash)
├── Queue: BullMQ or Inngest
└── Auth: Clerk or Auth.js

AI LAYER
├── LLM: Claude API (primary) + OpenAI (fallback)
├── Framework: LangChain or Vercel AI SDK
├── Vector DB: Pinecone or pgvector
├── Embeddings: OpenAI text-embedding-3-large
└── Observability: LangSmith or Braintrust

INFRASTRUCTURE
├── CI/CD: GitHub Actions
├── Monitoring: Sentry + Datadog
├── Analytics: PostHog
├── Feature Flags: LaunchDarkly or Statsig
└── Payments: Stripe
```

#### Architecture Decisions

| Decision | Recommendation | Why |
|----------|---------------|-----|
| Monolith vs Microservices | **Monolith first** | Startups need speed. Decompose later. |
| Serverless vs Server | **Serverless** for most things | Lower ops burden. Vercel + Lambda. |
| SQL vs NoSQL | **PostgreSQL** | Versatile. pgvector for embeddings. |
| REST vs GraphQL vs tRPC | **tRPC** for full-stack TS, **REST** for multi-client | Type safety with tRPC is a massive win. |
| Self-hosted vs Managed | **Managed everything** at start | Focus on product, not infrastructure. |

## 7. Go-to-Market for AI Products

### GTM Strategies

| Strategy | Description | Best For |
|----------|-------------|----------|
| **Developer-Led** | Open-source core → paid features. DevRel + docs + community. | AI infrastructure, dev tools |
| **Product-Led** | Free tier → self-serve upgrade. Low-touch. | AI SaaS, productivity tools |
| **Sales-Led** | Enterprise sales team. High-touch. | Enterprise AI, healthcare AI |
| **Community-Led** | Build community first → monetize later. | AI platforms, marketplaces |
| **API-First** | Ship an API → developers build on top → expand to enterprise | AI model providers, data platforms |
| **Bottom-Up SaaS** | Individual users adopt → teams adopt → company adopts | Collaboration AI tools |

### AI-Specific GTM Considerations

1. **Demo-ability matters** — AI products need to be shown, not described
2. **Trust is paramount** — Especially in healthcare, finance, legal
3. **Accuracy claims need proof** — Benchmark results, case studies
4. **Pricing is evolving** — Usage-based (tokens, API calls) is common for AI
5. **Data moats** — Your unique data advantage is your competitive moat
6. **Speed to value** — AI products that show value in < 5 minutes win
